häufig verwendete Befehle

  • git config --list
    Zeigt die Konfiguration an, zusammengefasst wird die Systemkonfiguration (--system, /etc/gitconfig), die Benutzerkonfiguration (--global, ~/.gitconfig) und die Repositorykonfiguration (<repository>/.git/config)

  • git config branch.master.rebase true
    Setzt einen Konfigurationseintrag in der Konfigurationsdatei des aktuellen Repositories. Die Schalter --system und --global modifizieren die System- bzw. Benutzerkonfiguration.

    Ergebnis in <repository>/.git/config:
    [branch "master"]
        rebase = true

    Folgende Befehle verwendet man nach der Installation zum Setzen von Benutzername und E-Mail-Adresse:
    git config --global user.name "Alexander Groß"
    git config --global user.email "agross@therightstuff.de"

    Mit Aliasen für häufig verwendete Befehle kann man sich einiges an Tipparbeit ersparen.
    git config --global alias.s       status
    git config --global alias.l       'log --oneline -10'
    git config --global alias.cm      'commit --message'
    git config --global alias.amend   'commit --amend --reuse-message=HEAD'
    git config --global alias.k       '!gitk --all &'
    git config --global alias.useless '!echo $PWD'

    Der Aufruf von git selbst lässt sich zusätzlich abkürzen:
    echo "alias g=git" >> ~/.bashrc

    Unter der Voraussetzung obiger Aliase zeigt g s den Status.

  • git init
    Initialisiert ein neues Git-Repository im aktuellen Verzeichnis.

  • git clone user@server:project.git
    Lädt das Repository vom Server per SSH in das Verzeichnis project herunter. Alternativ kann ein anderes Verzeichnis als letzter Parameter angegeben werden.

  • git add <path>
    Fügt eine neue oder geänderte Datei dem nächsten Commit (Index) hinzu. Pfade werden rekursiv durchlaufen. Alternativ kann git gui verwendet werden.

    Schalter:
    • --update, -u
      Fügt nur Dateien hinzu, die bereits unter Source Control sind
    • --all, -A
      Fügt alle Dateien hinzu
    • --patch, -p
      Fragt für jede Änderung, ob diese hinzugefügt werden soll. Ermöglicht partielle Commits wenn z. B. in einer Datei neben der Featureimplementierung ein davon unabhängiger Schreibfehler korrigiert wurde.
    • --interactive, -i
      Zeigt eine "Konsolen-UI" für das Hinzufügen von Dateien an.

  • git rm  <path>
    Entfernt die Datei/Pfad von der Festplatte und fügt sie als gelöscht dem nächsten Commit (Index) hinzu.

  • git commit
    Erzeugt einen neuen Commit auf Basis des Index.

    Schalter:
    • --message, -m
      Nimmt die Commitnachricht per Kommandozeile entgegen.
    • --all, -a
      Nimmt alle Dateien unter Source Control automatisch in den Commit auf (wirkt wie git add --update)
    • --amend
      Modifiziert den letzten Commit auf Basis des aktuellen Index. Kann verwendet werden, um vergessene Dateien nachträglich hinzuzufügen.       Vorsicht beim Modifizieren öffentlicher von Servern abgerufener Commits!
    • --reuse-message <commit>, -C <commit>
      Nutzt die Commitnachricht des durch <commit> spezifizierten Commits. Häufige Nutzung mit --amend, da die Commitnachricht nicht modifiziert werden muss:       git commit --amend --reuse-message=HEAD

  • git log <commit> <path>
    Zeigt die Historie eines Branches an. Ohne Parameter wird der aktuelle Branch/HEAD verwendet.
    Einschänkungen auf Dateien erfolgt per <Pfad>:  git log README 
    Andere Branches/Commits können per <commit> referenziert werden:  git log origin/master

    Schalter:
    • --oneline
      Zeigt eine Zeile pro Commit an. Darüber hinaus gibt es zig Optionen zum Modifizieren der Ausgabe.
    • --max-count=<n>, -<n>
      Zeigt die letzten <n> Commits.
    • --patch, -p
      Zeigt zum Logeintrag zusätzlich den Diff an. "Review" des letzten Commits:       git log -1 --patch

  • git push <remote> <branch>
    Überträgt die lokale Historie an einen Server (== "remote"). <remote> und <branch> sind optional, bei Weglassung werden alle mit Servern verbundene Branches (== "remote tracking branches") hochgeladen.

    Schalter:
    • --set-upstream, -u
      Koppelt den Branch mit dem Remote Branch und ermöglicht dadurch       git push/pull ohne weitere Parameter. Beispiel: git push origin master --set-upstream
    • --force, -f
      Git lehnt Übertragungen ab welche serverseitige Historie überschreibt. Mit diesem Parameter kann die Übertragung erzwungen werden, mit u. U. negativen Folgen für Kollegen die auf dem überschriebenen Zustand aufgesetzt haben.
    • --delete
      Löscht einen Branch aus dem Remote Repository. 

  • git pull <remote>
    Ruft Historie von einem Server ab. Der Standardservername (<remote>) ist "origin".

    git pull     ist ein Alias für git fetch (reiner Download) und, in Abhängigkeit der Konfiguration des Branches, git merge bzw. git rebase.

    git merge ist die Standardeinstellung. git rebase kann durch Konfiguration von git config branch.<name>.rebase true aktiviert werden. Die Konfigurationseinstellung core.autosetuprebase aktiviert Rebasing automatisch für alle nachfolgend erzeugten Branches.

  • git merge <branch>
    Führt <branch> und den aktuellen Branch zusammen.

    Ist die Historie beider Branches linear bzw. nicht divergent findet ein fast-forward Merge statt. Hierbei wird der HEAD verschoben (siehe auch     git reset) und es entsteht kein Merge Commit.

    Ist die Historie divergent entsteht ein Merge Commit.

    Schalter:
    • --no-ff
      Erzwingt einen Merge-Commit, auch wenn ein fast-forward möglich wäre.
    • --ff-only
      Erzwingt einen fast-forward Merge, bricht ab falls dieser nicht möglich ist.

  • git rebase
    Setzte des Abzweigpunktpunkt eines Branches neu. Interactive Rebasing erlaubt die Änderung von Historie durch Zusammenfügen, Löschen, Umsortieren und Modifizieren von Commits. Vorsicht beim Modifizieren öffentlicher von Servern abgerufener Commits!

    • git rebase master
      Setzte den Abzweigpunkt des aktuellen Branches auf den aktuellen Stand von master (jeder andere Commit ist ebenso möglich).
    • git rebase --interactive HEAD~5
      Erlaubt die Änderung der letzten fünf Commits im aktuellen Branch.
      • Während des Rebasings hält Git u. U. beim Anwenden von Commits an, z. B. wenn Konflikte entstehen oder der Benutzer das Bearbeiten eines Commits wünscht.
      • Ist die Bearbeitung abgeschlossen bzw. Konflikte gelöst wird der Rebase mit git rebase --continue fortgesetzt.
      • Abbrechen kann man ein Rebase immer mit git rebase --abort durch das der vorherige Zustand wiederhergestellt wird.
      • TortoiseGit kann aktuell nicht mit einem angehaltenen Rebase umgehen.

  • git reset <commit>
    Setzt den HEAD des aktuellen Branches auf einen beliebigen Commit.

    git reset HEAD <path> entfernt eine Datei/Pfad aus dem Index (der Liste zu committender Dateien). 

    Schalter:
    • --soft
      Verändert weder die Arbeitskopie noch den Index. Der Unterschied zwischen HEAD alt und HEAD neu wird durch git status als "Changes to be committed" ausgegeben.
    • --mixed
      Verändert nicht die Arbeitskopie aber den Index.       Der Unterschied zwischen HEAD alt und HEAD neu wird durch git status als "Changes not staged for commit" ausgegeben.
    • --hard
      Gleicht die Arbeitskopie an den Zustand von <commit> an. Der Inhalt des Index geht verloren.

  • git checkout
    Wechselt Branches und setzt geänderte Dateien wieder zurück (à la svn revert).

    • git checkout <path>
      Setzt geänderte Dateien unter <path> auf den aktuellen HEAD zurück.
    • git checkout -b <branch_name>
      Erzeugt und wechselt auf einen neuen Branch auf Basis des aktuellen HEAD. Geänderte Dateien in der Arbeitskopie werden übernommen.
    • git checkout <branch_name>
      Wechselt auf den Branch <branch_name> (à la       svn switch).

  • git branch
    Zeigt Entwicklungszweige an.

    Schalter:
    • -a
      Zeigt lokale und remote Branches an.
    • --verbose, -v
      Zeigt den aktuellsten Commit eines jeden Branches an.
    • -d, bzw. -D
      Löscht einen lokalen Branch. Sind die Commits des zu löschenden Branches nicht über einen anderen Branch erreichbar ("not fully merged") kann die Löschung mit -D erzwungen werden.
      Branches werden beim lokalen Löschen nicht vom Server entfernt. Hierfür verwenden Sie git push --delete <remote> <branch> oder       git-publish-branch.rb.

  • git diff
    Zeigt Veränderungen an Dateien in der Arbeitskopie gegen den Zustand in der Historie und Index an. git diff <from>..<to> zeigt die Veränderungen über Revisionen an. Beispiel: git diff master..feature

    Schalter:
    • --cached
      Difft den Index gegen die Historie. Nützlich um herauszufinden was committed wird.
    • --word-diff
      Zeigt Änderungen innerhalb der Zeile an. Nützlich bei minimalen Änderungen in langen Zeilen (z. B. Fließtext).
    • --ignore-all-space
      Ignoriert Änderungen an Einzügen, bspw. nach der Codeformatierung. Kombination aus       --ignore-space-at-eol und --ignore-space-change.

  • git mergetool
    Ruft das Tool zur Behebung von Mergekonflikten auf. Es können beliebige Tools verwendet werden, benötigt u. U. manuelle Konfiguration.

  • git reflog
    "git reflog will tell you where your head has been ... that sounded dirty. @chacon at #froscon"
    Ruft den "Papierkorb" von Git auf, in dem die letzten Aktionen im Repository aufgezeichnet sind. Kann verwendet werden, um unabsichtlich gelöschte Commits wiederherzustellen (siehe git reset und git cherry-pick).

  • git cherry-pick <commit>
    Wendet einen Patch auf die Arbeitskopie an. Kann z. B. genutzt werden, um Patches von anderen Branches ohne git merge oder git rebase auf den aktuellen Branch zu ziehen.

  • git stash
    Legt den Index und die Änderungen an Dateien unter Source Control "auf Halde".

    • git stash save <message>
      Erzeugt einen neuen Stash mit benutzerspezifischer Nachricht. Empfohlen, da die Übersicht bei Gits eigenen "WIP on <branch>: <letzter Commit>"-Nachrichten schnell verloren geht.
    • git stash list
      Listet den Stack an Stashes auf.
    • git stash pop
      Wendet den obersten Stash auf das Arbeitsverzeichnis an und verwirft ihn.
    • git stash apply
      Wendet den obersten Stash auf das Arbeitsverzeichnis an und verwirft ihn nicht.
